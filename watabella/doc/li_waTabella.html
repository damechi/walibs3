waTabella 3.0

 

waTabella 3.0

 

Classe per la gestione di una tabella con alimentazione proveniente da una query SQL o da un array associativo.

Features:

  • paginazione dei record
  • gestione predefinita delle azioni ricorrenti (vedi/modifica/elimina/nuovo)
  • definizione di un protocollo per il colloquio client/server per la gestione dei criteri di ordinamento e filtro dei record
  • output generato mediante foglio di stile XSLT
  • possibilità di data-editing
Requires:
  • waDb

Di per sè, la classe genera solamente un codice XML che viene dato in pasto ad una UI (User Interface); questa UI conterrà il foglio di stile XSLT che provvederà a renderizzare l'output e a fornire l'implementazione di tutte le procedure di controllo che agiranno lato client. Se non viene specificata una UI (cioè se non viene valorizzata la proprietà watabella::xslt) verrà utilizzata una UI di default (all'interno del package la directory: uis/wa_azioni_sx_default), che prevede un output XHTML 1.0 strict, l'implementazione delle azioni tramite bottoni e una gestione delle azioni tramite classi javascript (si veda al proposito la relativa documentazione UI).

Come si usa il package? Seguite questo esempio di codice PHP: 
<?php

// includete il package waTabella
include "watabella.inc.php";

// definite una query sql
$sql = "SELECT * FROM Ordini";

// create un oggetto di classe waTabella e associatevi la stringa sql
$table = new waTabella($sql, './file_configurazione_wadb.inc.php');

// definite le proprietà della tabella a vostro piacere
$table->titolo = "Tabella degli Ordini";
$table->paginaModulo = "modulo_ordini.php";

// aggiungete le azioni applicative che la tabella dovrà svolgere
$table->aggiungiAzione("Archivia", true, "Archivia ordine");

// definite le colonne della tabella
$table->aggiungiColonna("IDOrdine", 'ID', true, true, true, WATBL_ALLINEA_DX);
$table->aggiungiColonna("NoteOrdine", 'Note');

// mandate in output la tabella
if (!$table->caricaRighe())
	exit("Errore caricaRighe: " . 
			$table->righeDB->nrErrore() . " - " . 
			$table->righeDB->messaggioErrore());
$table->mostra();


?>

Naturalmente potete personalizzare molto meglio la vostra tabella e renderla molto più complessa e adatta alle vostre esigenze. Per scoprire come vi invitiamo a scorrere le descrizioni delle diverse proprietà che le classi vi mettono a disposizione.

Il protocollo di comunicazione client/server mediante HTTP/GET

Il client può inviare alla classe operante sul server alcuni comandi predefiniti; ovviamente l'xslt deve predisporre un meccanismo che permetta all'utente di operare queste selezioni. I criteri devono essere passati al server sotto forma di parametri HTTP/GET. Di seguito il dettaglio dei parametri previsti dal protocollo:
  • watbl_pg[nome_tabella]: la classe carica i record previsti alla pagina indicata da questo parametro (0-based)
  • watbl_oc[nome_tabella][indice]: la classe imposta l'ordinamento sulle colonne i cui nomi sono passati come parametri all'interno dell'array, in ordine di indice. L'ordinamento avviene a condizione che la colonna individuata abbia la proprietà ordina = true.
  • watbl_om[nome_tabella][indice]: la classe imposta l'ordinamento sulle colonne i cui nomi sono passati come parametri all'interno dell'array watbl_oc[nome_tabella][indice], secondo la modalità passata in questo parametro nel corrispondente indice, modalità che potrà valere ASC (default) o DESC
  • watbl_fc[nome_tabella][indice]: la classe imposta il filtro sulle colonne i cui nomi sono passati come parametri all'interno dell'array. Il filtro viene impostato a condizione che la colonna individuata abbia la proprietà filtra = true e che siano valorizzati anche i corrispondenti elementi indice dei parametri watbl_fm[nome_tabella][indice] e watbl_fv[nome_tabella][indice]
  • watbl_fm[nome_tabella][indice]: la classe imposta il filtro sulle colonne i cui nomi sono passati come parametri all'interno dell'array watbl_fc[nome_tabella][indice], secondo l'operatore passato in questo parametro nel corrispondente indice. Gli operatori riconosciuti dalla classe vengono passati anche all'XSLT tramite l'elemento XPATH /watabella/modi_filtro
  • watbl_fv[nome_tabella][indice]: la classe imposta il filtro sulle colonne i cui nomi sono passati come parametri all'interno dell'array watbl_fc[nome_tabella][indice], secondo il valore passato in questo parametro nel corrispondente indice. Il valore è atteso dalla classe nel formato waLibs (data: Y-m-d, ora: H:i:s, ecc.)
  • watbl_or[nome_tabella]: la classe imposta l'ordinamento sulla colonna il cui nome è passato come parametro, a condizione che questa abbia la proprietà ordina = true. Rende inattivi i parametri watbl_oc[nome_tabella]
  • watbl_orm[nome_tabella]: la classe imposta l'ordinamento sulla colonna il cui nome è passato dal parametro watbl_or[nome_tabella], secondo la modalità passata in questo parametro, che potrà valere ASC (default) o DESC
  • watbl_rr[nome_tabella]: la classe effettua una ricerca del testo dato all'interno di ogni colonna della tabella che abbia la proprietà filtra = true. Il filtro si somma (in AND logico) ad eventuali filtri impostati tramite il parametro watbl_fc[nome_tabella].
  • watbl_esporta_csv[nome_tabella]: la classe genera un file in formato CSV di tutte le righe che soddisfano le condizioni impostate ed invia il file al client
  • watbl_esporta_xls[nome_tabella]: la classe genera un file in formato XLS (MS-Office 2003) di tutte le righe che soddisfano le condizioni impostate ed invia il file al client.
    Necessita dei pacchetti PHPExcel o ExcelWriterXML oppure il pacchetto PEAR Spreadsheet_Excel_Writer (in include_path, oppure nella working directory dello script)
  • watbl_esporta_pdf[nome_tabella]: la classe genera un file in formato PDF di tutte le righe che soddisfano le condizioni impostate ed invia il file al client.
    Necessita del pacchetto FPDF (in include_path, oppure nella working directory dello script)

Editing mediante HTTP/POST

La classe è in grado di fornire un servizio di editing dei dati mostrati nella tabella. Per poterlo utlizzare, occorre ovviamente un XSLT costruito all'uopo (tra le UI di default si può utilizzare, anche a titolo di esempio, la UI uis/wa_azioni_sx_edit). Inoltre occorre istruire la classe in questo modo:
  • ogni colonna che deve essere editabile deve avere la proprietà inputTipo valorizzata
  • prima di caricare le righe dal DB con il metodo caricaRighe occorre chiamare il metodo leggiValoriIngresso e verificare tramite il metodo daAggiornare se effettivamente sono stati inviati dati mediante HTTP/POST (i dati di input devono sempre arrivare alla classe mediante questa modalità del protocollo) ; se la tabella ha effettivamente ricevuto una richiesta di aggiornamento da parte del client, questa va gestita mediante chiamata al metodo salva, oppure manualmente.
  • I dati ricevuti tramite HTTP/POST dal client possono essere organizzati come si ritiene più opportuno: devono comunque essere convertiti dall'XSLT in un flusso XML comprensibile dalla classe waApplicazione (si vedano gli esempi XML di input, intermedio e finale). Un solo dato deve sempre essere esplicitamente presente, ossia il nome della tabella che ha richiesto il submit, individuato dal parametro watabella_nome_tabella: il meccanismo di aggiornamento viene innescato solo se questo parametro corrisponde alla proprietà nome dell'oggetto waTabella istanziato

RPC mediante HTTP/POST

La classe è in grado di fornire un servizio di RPC; per innescare il meccanismo è necessario inviare tramite HTTP/POST i seguenti parametri:
  • watabella_nome_tabella questo parametro deve corrispondere alla proprietà nome dell'oggetto waTabella istanziato
  • watabella_rpc valorizzato a 1
  • watabella_funzione_rpc nome della funzione PHP che si intende richiamare; il nome può anche essere un metodo dell'oggetto applicazione se la proprietà è stata opportunamente valorizzata
  • watabella_dati_rpc array dei parametri che si intendono passare alla funzione/metodo di cui al punto precedente; i parametri saranno passati in ordine posizionale
Anche in questo caso, come nel precedente, dovrà essere invocato il metodo leggiValoriIngresso affinchè la classe possa accorgersi che le è giunto un input da gestire.

Il risultato della funzione/metodo PHP può essere di un qualunque tipo semplice oppure un array. Al client sarà inviato in risposta un flusso XML così definito:

  • watabella_esito_rpc può valere WATBL_RPC_OK oppure WATBL_RPC_KO
  • watabella_messaggio_rpc eventuale messaggio di errore
  • watabella_dati_rpc dati in risposta (valore ritornato dalla funzione/metodo). In caso di array ogni elemento sarà contenuto in un elemento XML di nome item

Editing in RPC mediante HTTP/POST

Combinando le funzionalità descritte nei due capoversi precedenti, è possibile innescare un ulteriore meccanismo: quello del consolidamento immediato su base dati di un editing inviato tramite RPC (tipicamente alla perdita del focus di un controllo). Tra le UI di default si può utilizzare, anche a titolo di esempio, la UI uis/wa_azioni_sx_quick_edit

Per innescare il meccanismo:

  • inviare i medesimi parametri previsti per la funzionalità RPC, ad eccezione del parametro watabella_dati_rpc
  • valorizzare il parametro watabella_funzione_rpc col valore speciale "watabella_rpc_aggiornamento_immediato"
  • passare i parametri da consolidare tramite HTTP/POST nello stesso modo in cui sarebbero inviati durante la funzionalità di editing: questi parametri saranno passati all'XSLT che provvederà a trasformarli nell'XML comprensibile dalla classe.
I dati così inviati saranno automaticamente consolidati su database mediante chaimata al metodo salva e sarà automaticamente inviata al client un esito di tipo RPC.
